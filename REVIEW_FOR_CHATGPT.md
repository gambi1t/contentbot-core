## 1. Контекст

**maksim-bot** — Telegram-бот (форк рабочего @panferovai_contentbot) под клиента
Максима: производит вертикальные ролики (Reels/Shorts) из идеи. Бренд `maksim`.
Прод: сервер `nox-maksim`, systemd-сервис, бот `@yumsunov_content_bot`. Ядро —
один файл `bot.py` (~18.5к строк, python-telegram-bot, async). Деплой через scp +
restart. Регресс — через Telethon (живой бот, реальные нажатия).

**Текущий пайплайн «идея → ролик» (работает, но рваный):**
идея (текст/голос) → генерация сценария (Claude) → утвердить → обложка (фон-фото +
текст) → озвучка (ElevenLabs-клон ИЛИ своё голосовое) → аватар (HeyGen, Avatar 3 /
Avatar IV) → авто-сборка монтажа (ffmpeg, аватар + B-roll/фото) → публикация.
Параллельно есть автономный B-roll (Claude Code на сервере пишет Remotion-вставки
и рендерит) и кросс-постинг (TG/IG/YT/VK).

**Деньги.** Шаг аватара (HeyGen) и автономный B-roll (Claude Code API) — реальные
деньги за каждый запуск. Озвучка (ElevenLabs) — тоже. Генерация сценария/обложки
(Claude) — копейки. Это критично для дизайна автопрогона (см. §4).

**Состояние сейчас.** Прогресс пайплайна живёт в `pending[user_id]` (dict в
памяти + дамп в `pending.json`). Notion-карточка — отдельный источник правды по
статусу (свойство Status: «Идеи | старт» / «Сценарий | озвучка» / «Аватар |
генерации» / «Готово к публикации»). `/start` чистит `pending` (это by design, но
ломает любой незавершённый прогон — см. §4.3).

## 3. Рабочий процесс с GitHub (РЕШЕНИЕ — проверь его)

Параллельно идёт инициатива **«конструктор»**: вынести бота в мультитенантный
движок, чтобы новый клиент = конфиг + ассеты, без правок кода (сейчас онбординг
клиента = ~40+ часов). Делает это отдельный разработчик (зовём **Тёма**, junior,
RFC-first, уже приступил, читает наш handoff-пакет в `docs/`).

Репо `contentbot-core`:
- **`main`** — база Тёмы для конструктора. Мы её НЕ трогаем.
- **`maksim-prod`** — снимок живого прода + человекочитаемый журнал
  `docs/PROD_CHANGELOG.md` (что/зачем/где-в-коде/как-тест). Каждый блок: дописываю
  журнал + пушу ветку. Тёма читает журнал и переносит фичи в свою структуру.

**Вопрос к тебе:** разумна ли эта схема (прод-фичи в отдельной ветке, конструктор
строится из `main`, синхронизация через журнал + diff), или мы программируем себе
боль слияния? Прод активно меняется (мы добавляем фичи Максиму), а Тёма
одновременно рефакторит ту же кодобазу в модули. Как правильно развести «живой
прод, который нельзя замораживать» и «рефакторинг в конструктор»?

## 4. ГЛАВНОЕ НА РЕВЬЮ — дизайн «спайна» пайплайна (строим завтра)

**Проблема.** Сейчас пайплайн рваный: после каждого шага пользователь сам ищет
следующую кнопку, иногда копипастит текст между шагами, на части переходов кнопка
«мертвеет» после рестарта. Цель — единый «позвоночник», который ведёт от идеи до
готового ролика с минимумом ручных действий, но без потери контроля и без лишних
трат.

Согласованный с владельцем дизайн (вариант 1 + каскад A), который надо
пересмотреть критически:

### 4.1 Автозапуск AI-аватара без копипасты
Из тезиса идеи бот сам запускает генерацию сценария (сейчас местами надо выделить
и скопировать текст вручную). Триггер прямо из карточки/идеи.

### 4.2 Автопереход по шагам (вариант 1)
После каждого «Утвердить» бот сам ведёт на следующий шаг:
сценарий → обложка → озвучка → аватар → сборка. На каждом шаге панель:
`[Дальше] [⏭ Пропустить] [📥 Скачать материалы] [✅ Завершить]`.
- «Пропустить» оставляем — ей пользуется SMM-команда (домонтаж в CapCut).
- «Скачать материалы» доступна на каждом шаге.

### 4.3 Состояние, привязанное к карточке (а не к `pending[user]`)
Прогресс хранить так, чтобы переживал рестарт бота И `/start`. Идея — ключевать по
`notion_page_id` (Notion как источник правды по статусу + локальный кэш).

### 4.4 Каскад пропусков (вариант A) + несколько планов ролика
Пропустил аватар → сборка идёт как B-roll-only монтаж (без аватар-дорожки).
Планы: (1) живой аватар, (2) селфи, (3) только B-roll.

## 5. Открытые вопросы — на что я прошу тебя ОСОБЕННО надавить

1. **State-machine: где хранить прогресс?** Варианты: (а) свойство в Notion
   (источник правды, видно в доске, но медленно + rate-limit + сеть может упасть
   посреди прогона); (б) локальный JSON по `notion_page_id` (быстро, но не видно
   человеку, и это ещё один файл-состояние рядом с `pending.json`); (в) гибрид
   (локальный кэш + периодическая запись статуса в Notion). Что выбрать и как не
   развести рассинхрон между двумя источниками?

2. **Деньги при автопрогоне.** Шаги аватара (HeyGen) и автономного B-roll — реальные
   деньги за запуск. «Автопереход» НЕ должен сам тратить деньги. Где правильно
   ставить обязательный ручной confirm? Достаточно ли «авто до денежного шага →
   стоп → явное подтверждение»? Не сделает ли это «авто» бессмысленным?

3. **`/start` чистит `pending`.** Если мы переносим состояние в карточку, надо ли
   вообще менять поведение `/start`, или просто перестать зависеть от `pending`
   для прогресса? Риск: пользователь жмёт `/start`, теряет контекст, но карточка
   жива — как корректно «возобновить с шага N»?

4. **Несколько планов (аватар/селфи/B-roll) в монолите 18.5к строк.** Как развести
   ветвления без комбинаторного взрыва `if/elif` в одном файле? Стоит ли с самого
   начала вынести спайн в отдельный модуль-оркестратор (state machine), даже до
   общего рефакторинга Тёмы?

5. **Связь с конструктором (САМОЕ важное).** Если мы строим спайн прямо в `bot.py`,
   мы наращиваем монолит, который Тёма параллельно пилит на модули → конфликт.
   Варианты: (а) строим спайн как отдельный модуль-оркестратор с чистым
   интерфейсом (тенант-агностичный), чтобы он лёг в конструктор почти как есть;
   (б) строим по-быстрому в `bot.py`, Тёма потом вынесет; (в) ждём структуру Тёмы
   и строим спайн уже в ней. Что правильно с точки зрения «не плодить техдолг и не
   блокировать ни прод, ни конструктор»?

6. **Объём на одну сессию.** Реалистично ли всё §4 за одну сфокусированную сессию
   (связка владелец + Claude, темп: 14-слайдовая презентация = 1 сессия; средняя
   фича с TDD = часть сессии)? Что вынести в первый срез, что отложить?
