# Процесс и правила работы

Этот файл — для владельца и разработчика: как ведём работу, какие гейты, что
можно и нельзя. Цель — получить сильный RFC и PoC, не сломав живой продукт и не
наплодив параллельных версий.

## Поток работы (гейты)

```
GitHub-репо + пакет контекста (этот набор файлов)
   │
   ▼
[Гейт 0]  Артём погружается, задаёт уточняющие вопросы письменно
   │
   ▼
RFC v1 (по 04_RFC_TEMPLATE)  ──►  AI-ревью + ревью владельца  ──►  правки  ──►  RFC «Agreed»
   │
   ▼
[Гейт 1]  Согласован подход. ТОЛЬКО ПОСЛЕ ЭТОГО — код.
   │
   ▼
PoC: один вынесенный модуль работает end-to-end на dev-инстансе
   │
   ▼
[Гейт 2]  Демо + ревью. Решаем, как строим дальше (уже мы).
```

**Правило №1: код не начинается до согласованного RFC.** Большие изменения
стоят дёшево на бумаге и дорого в коде. Это и есть способ не переписывать продукт
дважды.

## Definition of Done

**RFC готов, когда:**
- покрыты все разделы шаблона `04`;
- рассмотрено ≥2–3 альтернативы с честными плюсами/минусами;
- есть конкретная схема конфига тенанта + пример «как выглядел бы Максим»;
- есть план инкрементальной миграции без остановки прода и без второй версии;
- выбран модуль для PoC с обоснованием.

**PoC готов, когда:**
- один модуль вынесен по архитектуре из RFC;
- к нему подключается **новый (синтетический) тенант без правок кода** — это
  главный приёмочный тест;
- работает end-to-end на dev-инстансе, показано демо;
- не сломал живой бот (он вообще не затрагивается).

## Как не уехать не в ту ветку (важно для джуна)

Молодой разработчик + вайб-кодинг = риск, что AI уведёт реализацию не туда, а
человек не отвалидирует. Снимаем это процессом, а не контролем:

- **RFC-first** — сам формат заставляет думать до кода.
- **Маленький PoC** — один модуль, а не «весь конструктор». Ошибка дёшева.
- **Письменные апдейты** раз в 2–3 дня: что сделал, что застопорило, какой
  следующий шаг, какие развилки. Async, коротко.
- **Чекпоинты на развилках**: уперся в выбор с большими последствиями — пишешь
  владельцу/в чат с AI ДО того, как закодил, а не после.
- **Никаких «заодно переписал ещё вот это»** — scope строго в рамках RFC.

## Правило «один источник правды» (никаких параллельных версий)

- Живой бот = **baseline**. Все ветки — от него.
- Один основной репозиторий, feature-ветки, PR. Не «новый проект сбоку».
- Если RFC решит выносить ядро — оно остаётся **одним** ядром, общим для всех
  тенантов; тенанты — это данные, не копии кода.
- Любой «а давай заведу вторую систему и потом смержим» — красный флаг, стоп.

## Доступ (РЕШЕНО: изолированно, sandbox)

Решение владельца: **изолированный доступ, без прода.** (Не подлежит обсуждению
в рамках этого захода.)

Даём:
- **полный код** в чистом GitHub-репо (без `.env`, без секретов, история чистая);
- **dev-инстанс**: отдельный бот-токен от @BotFather, отдельная dev-Notion-база,
  **его собственные или жёстко лимитированные одноразовые** API-ключи;
- **синтетический тенант** (фейковые аватар/голос/соцсети) вместо данных клиентов.

НЕ даём: SSH на прод, прод-`.env`, реальные данные клиентов (Максим — клиент, его
персональные данные не передаются третьим лицам), боевые биллинговые ключи.

> Причина решения: метрические API-ключи = живые деньги; персональные данные
> клиента (Максим) = обязательство перед клиентом, их нельзя передавать третьим
> лицам; уход разработчика через ~6 недель = выданные секреты не отозвать «чисто»
> постфактум. Sandbox даёт ту же продуктивность без этих хвостов.

## Что отдаёт владелец на старте

1. GitHub-репо с кодом (очищенным от секретов).
2. Этот пакет контекста (`00`–`05`).
3. Доступ к dev-инстансу (по политике выше).
4. Контакт для вопросов + договорённость по ритму апдейтов.
