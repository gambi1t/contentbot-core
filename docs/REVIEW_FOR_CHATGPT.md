## 1. Что за продукт

Telegram-бот (Python) — фабрика коротких видео для личного бренда. Один человек +
ИИ делают полный цикл: идея → сценарий → говорящий ИИ-аватар (HeyGen) → озвучка
(ElevenLabs) → B-roll → монтаж (ffmpeg) → публикация (YouTube/VK/Instagram/TG).
Бот в проде, на нём идут реальные ролики и клиентские демо — **останавливать
нельзя**.

Сейчас два живых инстанса: бренд владельца и первый клиент. Второй — форк первого.

## 2. Проблема (от бизнеса)

Перенос продукта на нового клиента = **~40+ часов ручной работы**: лицо
(фото/видео-аватар), голос (клон), соцсети + OAuth, бизнес-факты, темп, рубрики,
площадки, промпты. При такой себестоимости продукт нельзя продавать пакетно.
Форк-подход дал второй симптом: при копировании в коде остались зашитые «следы»
прежнего аккаунта (чужие рубрики/площадки/имена/промпты), которые чистили вручную
по 18.5к-строчному монолиту.

## 3. Цель

**Конструктор клиентов:** запуск нового клиента = заполнить конфиг тенанта +
загрузить ассеты + чек-лист. **Без правок кода, за часы.** Приёмочный тест:
*«добавить клиента = один конфиг + ассеты, ноль изменений в .py»*.

## 4. Текущая архитектура (факты, сверено)

- **Монолит `bot.py` ~18 490 строк** + модули: `video_assembler.py` (1702),
  `crosspost.py` (1254), `launch_monitor.py` (1253), `tg_post_writer.py` (948),
  `instagram_dm.py` (596), `subtitle_burner.py` (541), `idea_generator.py` (410),
  `auto_broll.py` (315). Пакеты: `billing/` (~2700), `carousel/` (2243),
  `broll/` (937). Всего ~30к строк.
- **Мультитенантность сегодня** размазана по: dict `BRANDS` в коде (резолв
  `_resolve_brand_cfg` / ContextVar активного бренда), `.env` на инстанс, внешние
  промпт-файлы (`*_maksim.txt`), схема Notion клиента, OAuth-токены, ассеты на
  диске — **и зашитые литералы в монолите**. Уже есть ~40 точек бренд-ветвления.
- **FSM состояний** — на словаре `pending` + `pending.json` (не БД).
- **Деплой**: systemd-сервис + свой `.env` на инстанс; отдельный сервер/форк на
  клиента; dev/staging-контура нет.

## 5. Что варьируется на клиента (модель тенанта — собрано эмпирически)

Идентичность (имя/био/тон/гардероб) · аватар (HeyGen id, движок) · голос
(ElevenLabs id, модель) · соцсети (handle/каналы + OAuth-токены) · контент-
таксономия (Notion DB id, рубрики + правила классификации, площадки, форматы,
статусы — **имена опций должны совпадать со схемой Notion клиента**) · промпты
(сценарий/обложка/идея/посты/классификатор) · дефолты сборки · **AI-провайдер**
(сейчас зашит на Anthropic — для РФ-контура нужен сменный GigaChat/YandexGPT,
152-ФЗ) · биллинг/тариф/баланс.

## 6. Принятые решения (и обоснование)

1. **База = более продвинутый форк** (тот, где уже начат вынос тенанта). Аудит
   4 агентов показал: он **надмножество** второго (у второго нет ни одной фичи,
   которой нет в коде первого; то, что «казалось только там» — скрыто
   бренд-гейтом / не настроено / выключено). Второй форк ≈ «бренд default».
2. **Инкрементальный рефактор, НЕ rewrite с нуля.** Бот живой и приносит деньги/
   демо. Rewrite = риск двух параллельных версий (главный страх) + месяцы без
   фич.
3. **RFC-first.** Сначала проектный документ с альтернативами (без кода) →
   согласование → потом PoC одного модуля. Реализацию конструктора целиком в
   этот заход не делаем.
4. **Форма продукта (бот / Telegram Mini App / веб) — открытый вопрос для RFC.**
   Лин владельца — Mini App (dashboard + flow + табы, презентабельнее).
5. **Биллинг/баланс — в скоупе как тенант-фича, но пока выключен** (`BILLING_
   ENABLED=0`). MVP в коде есть (меню с балансом, тарифы self/full, списание,
   админка), но не обкатан и **без платёжного шлюза** (пополнение ручное).
6. **Исполнитель** — джуниор (сильный, вайб-кодит в Cursor, Python+Go), но через
   ~6 недель уходит. Поэтому его вклад = RFC + PoC, доступ изолированный (sandbox,
   без прода и боевых секретов).

## 7. План

`репо + контекст → RFC (с альтернативами) → ревью (владелец + AI) → PoC одного
вынесенного модуля → демо → дальше строим сами`. Гейт: код не начинается до
согласованного RFC. Guardrail: один источник правды, никаких параллельных версий.

## 8. На что особенно нужен твой критический взгляд

1. **Инкрементальный рефактор vs rewrite.** Для монолита 18.5к строк → чистая
   мультитенантность: реально ли вынимать модули из живого кода без «второй
   системы», или мы недооцениваем боль? Где подводные камни именно при
   инкрементальном пути?
2. **Полнота модели тенанта (раздел 5).** Каких измерений мультитенантности не
   хватает? (изоляция данных, секрет-менеджмент, отдельная БД на тенанта,
   наблюдаемость/логи на тенанта, лимиты/квоты, миграции схемы Notion, версии
   промптов, фичефлаги на тенанта?)
3. **«Конфиг + ассеты, ноль кода» — реалистично ли** для этого кода, и какие
   части сопротивляются такой декларативности сильнее всего?
4. **Форма: Telegram Mini App vs чат-бот vs веб.** С учётом цели (быстрый
   онбординг + презентабельность + та же бэкенд-логика) — что бы выбрал и почему?
   Какова реальная цена Mini App поверх существующего бэкенда?
5. **Резидентность данных (152-ФЗ).** Как чисто спроектировать **сменный
   AI-провайдер на уровне тенанта** (Anthropic/OpenAI vs GigaChat/YandexGPT),
   чтобы не плодить ветвления по всему коду?
6. **Состояние (FSM на `pending.json`).** Это слабое место для мультитенанта?
   Стоит ли в рамках конструктора уходить на БД-состояние, или это вне скоупа?
7. **Скоуп для джуна за 6 недель** (RFC + PoC одного модуля): адекватно? Какой
   модуль выбрать для PoC, чтобы он лучше всего доказал/опроверг подход
   «конфиг-driven тенант»?
8. **Риски, которые мы не видим.** Что чаще всего убивает такие
   «монолит → мультитенант-платформа» проекты?
