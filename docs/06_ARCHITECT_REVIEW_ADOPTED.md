# Внешнее архитектурное ревью — что принимаем (с CTO-скоупингом)

> Внешний старший архитектор (ChatGPT) разобрал `REVIEW_FOR_CHATGPT.md`. Ревью
> сильное и **подтвердило направление**. Здесь — что из него берём в работу и,
> главное, **как ограничиваем скоуп**, чтобы джун за 6 недель не построил
> «платформу на полгода». Это слой поверх `01`–`05`, не замена.

## Вердикт ревью (согласен)
- **Инкрементальный рефактор — да, rewrite — нет.** Подтверждено независимо.
- **Главный сдвиг рамки:** цель не «модульная система», а **tenant runtime** —
  «клиент = валидируемый конфиг + ассеты + секреты + сервисы, через который
  проходит любой сценарий». Бренд сегодня — не сущность, а россыпь условностей
  (`BRANDS`, `.env`, промпт-файлы, ContextVar, литералы). Просто «разнести по
  папкам» = тот же монолит в папках. **Принимаем эту рамку.**
- **PoC = вертикальный срез, не «вынесенный файл».** Сходится с нашим выбором:
  PoC на генерации **TG-поста/сценарного текста** (`tg_post_writer`-сценарий) —
  он уже частично бренд-зависим, трогает промпт/тон/провайдера/рубрики/output,
  но НЕ тянет ffmpeg и OAuth. Лучший кандидат.

## Что ДОПОЛНИТЕЛЬНО берём в модель тенанта (в `03`)
Ревью верно указало пробелы. Добавляем в проектирование тенант-контракта:
- `tenant_id` — короткий, **immutable**, безопасный для путей/логов (НЕ имя клиента).
- `schema_version`, `status` (active/paused/archived), `timezone/locale`.
- `data_region` + `compliance_mode` (none / ru_strict) — для 152-ФЗ.
- **Доступы**: admins / operators / allowed_chats (иначе оператор одного клиента
  дотянется до другого).
- **Secret refs**, а не токены в конфиге (`token_secret_ref: ...`).
- **Storage namespace на тенанта**: `/storage/{tenant_id}/...`, запрет общего `tmp`.
- **Provider policy на задачу** (не «какая модель», а политика: provider+model+
  prompt_pack per task + запрет иностранных провайдеров/PII для ru_strict).
- **Prompt pack с версией** (не россыпь `*_maksim.txt`); версия пишется в job metadata.
- **Notion mapping + валидатор** (свойства/опции → препрофлайт-проверка до запуска).
- **Feature flags** на тенанта (вместо `if brand == ...`).
- **Cost ledger** (событийная модель затрат) отдельно от «баланса».
- **Observability**: структурные логи с `tenant_id` / `job_id` / `step`.
- **Consent/retention** для лица/голоса/аватара (HeyGen/ElevenLabs — чувствительная зона).

## Архитектурные риски, которые ревью вскрыло (учесть в RFC)
- **ContextVar протекает** через async/фоновые задачи/коллбэки/ретраи/`pending.json`.
  → `tenant_id` обязан быть в payload/state каждого job-а; ContextVar — только
  удобный accessor, НЕ единственный источник истины. (У нас уже есть `_brand_ctx`
  ContextVar — это ровно та ловушка.)
- **Import-time side effects** убивают мультитенантность: у нас в `bot.py`
  Anthropic-клиент и промпты создаются/читаются НА ИМПОРТЕ (выбор тенанта — до
  запроса). → запретить module-global API-клиентов и загрузку промптов на импорте.
- **Резать ответственность, а не файлы**: единица выноса — порт/адаптер
  (`PromptService`, `LLMRouter`/`ProviderAdapter`, `AssetStore`, `StateStore`,
  `NotionGateway`, `SocialPublisher`), не «кусок из bot.py».

## ⚠️ CTO scope-guard: «проектируем под — строим сейчас»
Ревью очень полное — и в этом риск: если строить ВСЁ перечисленное, получим ту
самую «платформу на полгода», от которой ревью же и предостерегает (риск 9.10).
Поэтому жёстко делим:

**RFC ПРОЕКТИРУЕТ (на бумаге, весь контракт выше).**

**6-недельный PoC СТРОИТ только тонкий срез:**
1. `TenantConfig` (Pydantic-схема) + конфиги для 2 реальных клиентов + 1 фейковый.
2. `PromptService` (рендер промптов из prompt pack тенанта).
3. `LLMRouter` + **ровно 2 адаптера**: `AnthropicProvider` + `MockProvider`.
   (GigaChat/YandexGPT — интерфейс заложить, адаптеры писать позже, под реального
   РФ-клиента.)
4. PoC-сценарий: tenant-aware генерация TG-поста.
5. `validate_tenant.py` (CLI: схема + Notion preflight + наличие промптов).
6. Compatibility-слой, чтобы текущий прод не сломался.

**НЕ строим в этот заход** (проектируем, откладываем): cost ledger, observability-
платформу, quotas, `DbStateStore`, Mini App, billing gateway, провайдеры под РФ,
вынос `video_assembler`/OAuth. StateStore — вводим как **абстракцию** с
`JsonStateStore` (совместимость), миграцию в БД — после 3-го платящего клиента.

## Гейты приёмки PoC (взято из ревью, согласовано)
- **Гейт 1:** третий (фейковый) тенант добавляется **без правок `.py`** — иначе
  PoC не засчитан.
- **Гейт 2:** ошибка в tenant config ловится **валидатором ДО** генерации/публикации,
  а не в рантайме.
- **Гейт 3:** в новом коде появился `if tenant_id == ...` → откат, архитектура
  деградировала.
- Один код → разный вывод для 2 реальных + 1 фейкового тенанта; provider и prompt
  pack берутся из конфига; есть `MockProvider` и dry-run; логи с `tenant_id`.

## Где НЕ соглашаюсь / уточняю ревью
- **SQLite vs Postgres:** ревью склоняется к Postgres при фоновых воркерах. Для
  текущего одно-сервер/один-процесс инстанса — это преждевременная инфра. Держим
  state за `StateStore`, бэкенд (JSON→SQLite→Postgres) меняем по мере роста.
- **Объём провайдер-слоя:** capability-matrix + policy-guard + 4 адаптера — это
  дизайн-цель, НЕ PoC. В PoC — интерфейс + Anthropic + Mock.
- **Мы дальше гринфилда, чем кажется ревью:** существующий `BRANDS` +
  `_resolve_brand_cfg` + уже бренд-зависимые `_build_structure_prompt` /
  `_brand_rubrics` / `tg_post_writer._get_system_prompt` / `gen_description` —
  это **зачаток TenantRuntime**. Путь = эволюция `_resolve_brand_cfg(brand)` →
  `TenantRuntime`, а ~40 точек бренд-ветвления = готовая «tenant surface map».

## Revised roadmap (свёрнутый, под наши силы)
0. **Freeze:** база выбрана (этот форк), второй форк = бренд `default`/tenant.
   Запрет новых бренд-литералов (grep-чек).
1. **Tenant Surface Map** — добить grep-инвентарь (литералы / `.env` / промпты /
   OAuth / Notion-опции / пути ассетов / точки ContextVar+BRANDS). У нас уже ~80%
   собрано в де-форке.
2. **Tenant Contract RFC** (полный контракт выше) + Pydantic-схема + 2 реальных
   `tenant.yaml` + demo + validator CLI.
3. **PromptService + LLMRouter(Anthropic+Mock)**.
4. **PoC vertical slice** (TG-пост) + 3 гейта.
5. Гейт-решение: масштабируем паттерн или правим. **Только потом** — StateStore→БД,
   Notion-валидатор глубже, админка-в-боте → Mini App, cost ledger → billing.
