# Параллельные сессии: git worktree + дисциплина core/style

> Операционный спек (sibling к `DEPLOY_RUNBOOK.md`). Источник правды для ОБЕИХ
> Claude-сессий, работающих на одном ядре `maksim-prod` (тенанты maksim «Нолан»
> и panferov «Илон»). Прошёл efficiency-фильтр + adversarial CTO-критику
> (ресёрч 2025-26 + проверка кода). Версия 1 — 2026-06-24.

## Зачем (проблема, которую закрываем)

Две Claude-сессии сидят в ОДНОМ рабочем дереве на ветке `maksim-prod` с общим
HEAD. Симптомы (наблюдались вживую):
- **Коммит-хаос** — коммиты одной сессии ложатся fast-forward поверх другой.
- **Франкенштейн-деплои** — незакоммиченный WIP одной сессии уезжает `scp`-деплоем
  другой (кейс утечки `montage_brand_name`).

Корень — общее рабочее дерево, а не код. Лечится разделением деревьев + одним
гейтом деплоя.

## Модель (РЕШЕНИЕ)

**git worktree, 2 ПОСТОЯННЫХ дерева** (не дерево-на-фичу):

- `maksim-prod` = единственная integration-ветка, **в ней никто не сидит** (только
  мержи входят).
- Каждая сессия — в СВОЁМ постоянном дереве, на своей рабочей ветке.
- **Git физически запрещает двум деревьям держать одну ветку** (`branch already
  checked out`) → коммит-хаос исключён конструкцией, а не дисциплиной.
- venv/node_modules ставятся ОДИН раз на дерево (постоянные деревья → нет
  bootstrap-налога на каждую фичу).

Почему 2 постоянных дерева, а не дерево-на-фичу: Remotion `node_modules` + Python
`venv` — gitignored, в новое дерево не копируются. Дерево-на-фичу = переустановка
на каждую фичу (10-мин налог → откат к статус-кво). Постоянное дерево платит
bootstrap один раз.

## Раскладка деревьев (ПРЕДЛОЖЕНО — подтвердить)

| Дерево | Сессия | Рабочая ветка |
|---|---|---|
| `D:\AI\maksim-bot` (остаётся) | maksim / бот «Нолан» | `work/maksim` (или `feat/<…>`) |
| `D:\AI\maksim-bot-panferov` (новое) | panferov / бот «Илон» | `work/panferov` (или `feat/<…>`) |

Рекомендация — maksim остаётся в `D:\AI\maksim-bot` (так зашито в памяти и
деплой-скиллах; меньше переделок путей), panferov уезжает в новое дерево.

## Bootstrap нового дерева (один раз)

```powershell
# из основного дерева:
git worktree add D:\AI\maksim-bot-panferov -b work/panferov maksim-prod
cd D:\AI\maksim-bot-panferov
python -m venv .venv ; .\.venv\Scripts\pip install -r requirements.txt
npm install            # Remotion-зависимости (gitignored, в дерево не копируются)
# скопировать gitignored секреты/.env, которых нет в git, из основного дерева
```

⚠️ **НЕ симлинкать/junction'ить `node_modules` на Windows** — npm плодит junctions,
Node-резолв спотыкается, ветки расходятся по версиям пакетов. Только отдельная
установка. (Стоит вынести в `scripts/setup-worktree.ps1` — идемпотентный bootstrap.)

## Дневной git-флоу

1. **Правишь ЯДРО** (tenant-нейтральный код — напр. `_scene_count_for_script` в
   `hyperframes_broll.py`) → коммит → **в тот же день мерж в `maksim-prod` + push**.
2. **Вторая сессия** раз в день: `git fetch && git merge origin/maksim-prod` в свою
   ветку → подтянула чужое ядро. Ветки короткие (1-2 дня) → конфликтов мало.
3. **Правишь СТИЛЬ** (`style_contract.<tenant>.json`) → коммит остаётся в своей
   ветке, в ядро не течёт (физически не в core-коммите).
4. **Деплой** — только из чистого дерева (см. гейт ниже).

**Как ядро доезжает до ОБОИХ:** ядро закоммичено в `maksim-prod` → оба тенанта на
одном ядре (форка нет) → при деплое каждого бота улучшение уже там. Не «добавляешь
везде» — меняешь в одном общем месте.

**Как стиль остаётся РАЗДЕЛЬНЫМ:** см. `03_TENANT_MODEL.md` + `style_contract.py`.
Палитра/персона резолвятся по `tenant.active_tenant_id()` из
`style_contract.<tenant>.json` (panferov — свой контракт, maksim — дефолтный).
Один алгоритм → разные цвета по тенанту, разница только в конфиге.

## Правило core/style (закон)

> Ядро (`hyperframes_broll.py` и пр. tenant-нейтральные модули) НЕ содержит
> бренд-литералов (цвета, темы, имена брендов). Палитра/персона приходят ТОЛЬКО
> через `tenant.active_tenant_id()` + `style_contract.<tenant>.json`. Тема — это
> данные, не код. Не плодить `hyperframes_broll_<tenant>.py`.

Коммит-сообщение помечает scope: `core:` / `style(<tenant>):` / `mix:`. Гарантию
даёт не дробление коммитов (AI-сессии его обходят), а **страж по путям** (ниже).

## Гейт деплоя (главный анти-франкенштейн) — ВНЕДРЁН в deploy-maksim-bot

`deploy-maksim-bot`, Шаг 1.5: перед scp проверяем, что КАЖДЫЙ файл из списка
деплоя закоммичен (без WIP):

```bash
cd D:\AI\maksim-bot && git status --porcelain -- <file1> <file2> ...
# непусто → грязный файл → деплой запрещён, сначала закоммить
```

Деплоим воспроизводимый коммит, а не живую папку → незакоммиченный WIP (в т.ч.
чужой из параллельной сессии) физически не уезжает на сервер. Проверяем именно
файлы списка, а не всё дерево — untracked-черновики не должны блокировать деплой.
Один гейт: без `git archive` и без жёсткого `HEAD==maksim-prod` (последний
конфликтует с feature-ветками этой модели). Дисциплина «влить ядро в maksim-prod»
живёт в дневном флоу выше, не в гейте. (То же внедрить в `deploy-content-bot`.)

## Страж от протечки бренда в ядро (один pytest)

`tests/test_core_no_brand_leak.py` — грепает tenant-нейтральные модули ядра на
бренд-литералы: hex-цвета `#[0-9A-Fa-f]{3,6}`, бренд-слова и значения `accent` из
ОБОИХ `style_contract.*.json`. Матч в ядре → fail с номером строки. Источник правды
для запрещённого — сами контракты (добавление тенанта не требует правки стража).

⚠️ Текущий `check_forbidden_in_html` односторонний — сканирует сгенерированный HTML,
а НЕ код-билдеры промпта. Новый страж должен скан**ировать сам код** (`_build_*`).

Известный долг (P2, НЕ горит): `hyperframes_broll.py:~1357` хардкодит
`accent оранжевый #FF5722` в `_build_prompt`. Проверено: вызывается ТОЛЬКО под
`if legacy_build:` (env `HF_LEGACY_BUILD=1`, по умолчанию выкл). Боевые билдеры
`_build_scene_prompt` / `_build_scene_singleshot_prompt` уже тянут стиль из
контракта per-tenant. Т.е. это легаси-гигиена, не боевая протечка. Страж сделает
строку красной → почистить → зелёный замок навсегда (TDD: red → fix → green).

## Что НЕ делаем (отвергнутые альтернативы)

- **2 сессии в одном дереве** — статус-кво, источник проблемы.
- **Два отдельных `git clone`** — дублирует `.git`, обмен ядром только через
  push/fetch вручную → расходятся.
- **Per-tenant долгоживущие ветки + cherry-pick ядра** — ядро переносить руками
  дважды, дивергенция.
- **Дерево-на-фичу** — bootstrap-налог на каждую фичу.
- **Полный Gitflow / pre-commit-хук + git-archive-деплой + ArchUnit** —
  оверинжиниринг для команды из 2 человек. Один гейт + один pytest достаточно.
- **Внешнее ревью ChatGPT этой структуры — НЕ нужно.** Паттерн стандартный;
  реальные риски — в фактах о репо (origin? легаси-путь?), которых внешний ИИ не
  видит. Оба факта проверены локально (origin есть; 1357 — легаси).
