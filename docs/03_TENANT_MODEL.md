# Модель тенанта: что меняется от клиента к клиенту

> Это эмпирический список, собранный на **реальном переносе** продукта на
> клиента (Максим) и последующей чистке зашитых следов. То есть это не теория —
> это ровно те сущности, которые пришлось трогать руками. Конструктор должен
> вынести ВСЁ это в данные тенанта (конфиг + ассеты + внешние интеграции),
> чтобы запуск клиента не требовал правок кода.

Сгруппировано по типу. В скобках — где это живёт сейчас (чтобы понимать масштаб
выноса).

## 1. Идентичность

- Имя бренда / человека, дескрипшен, био автора *(BRANDS dict + зашито в промптах)*
- Тон, темп подачи, гардеробные режимы (под формат: outdoor/indoor/зима) *(BRANDS)*
- Бизнес-факты (чем занимается, ключевые цифры, продукты) *(внешние промпт-файлы)*

## 2. Лицо и голос (ассеты + внешние ID)

- **HeyGen**: `avatar_id`, набор «луков», версия движка (Avatar IV vs III) *(BRANDS)*
- **ElevenLabs**: `voice_id`, модель (v2/v3), настройки стабильности/стиля *(BRANDS)*
- Исходники для регистрации: фото для аватара, аудио для клона голоса *(ручной процесс)*
- Фото-банк / B-roll библиотека под бренд *(папки на сервере)*

## 3. Соцсети и публикация (интеграции + OAuth)

- Telegram-канал: handle, display-name, chat_id *(BRANDS + .env `TELEGRAM_CHANNEL_ID`)*
- Аккаунты IG / YouTube / VK / TikTok + их **OAuth-токены** *(.env + crosspost.py)*
- Какие площадки активны у клиента *(BRANDS `platforms`)*
- Мастер-пост / ссылка для DM-автоответа *(.env `DEFAULT_DM_REPLY_URL`)*
- Блок «об авторе» и CTA (подпись под гайдами/постами) *(.env `AUTHOR_*`)*
- База гайдов/лид-магнитов + публичный notion.site host *(.env `NOTION_GUIDES_DB_ID`, `NOTION_GUIDES_PUBLIC_HOST` — у каждого воркспейса свой subdomain!)*

## 4. Контент-таксономия (привязана к базе Notion клиента)

- **ID базы Notion** клиента *(.env `NOTION_DATABASE_ID`)*
- **Рубрики** — список + правила классификации + контекст для LLM *(BRANDS, бренд-зависимо)*
- **Площадки** и **«Опубликовано на»** — имена опций multi_select *(должны совпадать со схемой Notion!)*
- **Форматы** — опции *(должны совпадать со схемой Notion)*
- **Статусы** + тип свойства (`select` vs встроенный `status`) *(BRANDS)*
- Имя свойства рубрики (у разных баз бывает с пробелом/без) *(BRANDS `notion_rubric_property`)*

> ⚠️ Урок: коды площадок/рубрик в коде ДОЛЖНЫ совпадать со значениями
> multi_select в Notion-базе клиента, иначе Notion молча создаёт «левые» опции
> и засоряет базу. Это классический источник «следов форка».

## 5. Промпты (генеративное ядро)

- Промпт сценария, промпт обложки, промпт идей *(внешние `*_maksim.txt` файлы)*
- Системный промпт TG-постов *(уже вынесен: `SYSTEM_PROMPT_<brand>` в tg_post_writer)*
- Промпт-классификатор «идея → рубрика» + few-shot примеры *(был зашит, вынесён в де-форке)*
- Few-shot примеры в генераторах (легко протекают чужим брендом!) *(зашито)*

## 6. Сборка и дефолты

- Layout монтажа по умолчанию (pro/smart/…), субтитры вкл/выкл *(BRANDS)*
- Параметры smart-mix, авто-обрезка клипов *(BRANDS)*

## 7. Инфраструктура и провайдеры

- Все секреты/ключи *(.env)*
- **AI-провайдер** (Anthropic/OpenAI vs РФ-совместимый GigaChat/YandexGPT) —
  сейчас зашит на Anthropic. Для РФ-контура (Max, 152-ФЗ) нужен сменный провайдер
  **на уровне тенанта**. Этого слоя сейчас НЕТ — заложи в RFC.
- Биллинг вкл/выкл и тариф *(billing/, env)*

## 8. Биллинг / баланс (полноценная тенант-фича — пока ВЫКЛЮЧЕНА)

Решение владельца: баланс/биллинг **входит в скоуп конструктора** как тенант-фича,
но пока остаётся выключенным и дорабатывается позже. В коде уже есть рабочий MVP
(пакет `billing/` ~2700 строк): клиентское меню с балансом, история операций,
запрос пополнения, режимы тарифа (self/full), списание за ролик, биллинг-гейт,
админ-панель (клиенты, пополнение, корректировка, отчёты). Управляется флагом
`BILLING_ENABLED` (сейчас `0` везде).

Что из этого — данные тенанта (для конструктора):
- Включён ли биллинг для этого клиента *(env `BILLING_ENABLED`)*
- Тариф/режим по умолчанию (self 350 ₽ / full 2500 ₽) и цены *(billing config/БД)*
- Баланс и история операций клиента *(billing БД, per-client)*
- Контакт поддержки *(env `SUPPORT_CONTACT`)*, идентификатор инстанса *(env `BOT_INSTANCE`)*
- Тип главного меню: «владелец без баланса» (`ilan_main`) vs «клиент с балансом»
  (`client_main`) — это уже готовый мультитенантный паттерн в коде.

Что НЕ доделано (заложить в RFC, но не сейчас): **платёжный шлюз** (сейчас
пополнение ручное — клиент запрашивает, админ зачисляет); биллинг **не обкатан**
на живом платящем клиенте.

## Вывод для RFC

«Тенант» сегодня размазан по: `BRANDS` (dict в коде), `.env`, внешним
промпт-файлам, схеме Notion, OAuth-токенам, ассетам на диске **и зашитым
литералам в монолите**. Конструктор должен свести это к:

1. **Декларативный конфиг тенанта** (один источник: что за бренд, какие ID,
   какие промпты, какая таксономия, какой AI-провайдер).
2. **Хранилище ассетов** тенанта (фото/аудио/видео/банк B-roll).
3. **Реестр интеграций** (токены/OAuth, изолированно и безопасно).
4. **Слой провайдеров** (AI, аватар, голос, площадки — сменяемые).

Ключевой тест архитектуры: *«добавить нового клиента = создать один конфиг +
загрузить ассеты + пройти чек-лист, ноль изменений в коде»*. Если для нового
клиента надо тронуть `.py` — конструктор не готов.
